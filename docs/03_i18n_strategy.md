# 03 - 多語系優化與自動化規格書

## 1. 痛點分析 (Pain Points)
*   **手動修正繁瑣：** 需要修改 JSON 檔後再重新部署。
*   **翻譯不一致：** 不同頁面對同一專有名詞翻譯不同。
*   **開發阻塞：** 工程師需等待翻譯完成才能推進。

## 2. 核心架構 (Core i18n Engine)
我們將採取 **「動靜分離」** 與 **「動態更新」** 策略。

### 2.1 靜態翻譯 (Static UI)
*   **框架：** `react-i18next`。
*   **儲存：** 前端 `/public/locales` 目錄下的 JSON 檔。
*   **自動化工具：** 
    *   **i18n-ally：** 提供 IDE 內嵌翻譯與重複 key 偵測。
    *   **提取工具：** `i18next-scanner` 自動掃描代碼中的 `t()` 函數，產出待翻譯清單。

### 2.2 動態內容 (Dynamic Data)
*   **資料庫存儲：**
    *   所有具備多語系需求的欄位 (如課程描述)，在資料庫中統一存儲為 JSON 物件：
    ```json
    {
      "zh_TW": "初階滑雪課",
      "en_US": "Beginner Ski Course",
      "ja_JP": "初心者スキーコース",
      "yue_HK": "初階滑雪堂"
    }
    ```
*   **優點：** 擴充新語言只需在 JSON 加 key，無需改 Schema。

## 3. 自動化翻譯流程 (Optimization Flow)
1.  **管理員修正介面：** 後台提供翻譯管理頁面，讓行政人員直接編輯資料庫中的翻譯內容。
2.  **即時套用：** 編輯後前端立即生效，無需重啟伺服器或前端。
3.  **AI 輔助：** 串接 OpenAI / DeepL API 進行預翻譯，行政人員僅需進行最後「校正 (Calibration)」，提升 80% 速度。

## 4. 前端檢測與規範
*   **強型別 Key：** 使用 TypeScript 定義翻譯 key，避免拼錯字。
*   **缺失警告：** 測試環境下，若有 key 缺失翻譯，介面顯示 `[MISSING: KEY_NAME]` 高亮提示。
